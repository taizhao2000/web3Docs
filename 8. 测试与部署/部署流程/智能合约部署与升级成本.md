> **预计阅读时间**：25 分钟

## 1. 部署≠执行：一个关键澄清

很多人以为"部署合约"就是"全网节点运行你的代码"，这是误解。

```
部署合约时（发送 deployment 交易）：

  节点做什么？
    1. 验证交易签名 ✅
    2. 执行构造函数（constructor）✅（只跑一次，用于初始化状态）
    3. 把字节码存到链上 ✅
    4. 给合约分配一个地址 ✅
    5. ──────────────────────
    6. 此后合约的代码就躺在链上，没人主动去执行它

  类比：你把一本操作手册放到图书馆的书架上
        图书馆不会把手册里的步骤执行一遍
        只是把它存好，等有人来读的时候再说

调用合约时（发送交易到合约地址）：

  节点做什么？
    1. 从链上取出合约的字节码
    2. 在本地 EVM 中执行
    3. 验证执行结果是否与其他节点一致
    4. 如果一致 → 状态变更被确认
    5. ──────────────────────
    6. 全网所有全节点都在本地执行同样的代码

  类比：有人来图书馆按手册操作
        所有图书馆都要同步执行，确保结果一致
```

**核心结论：不是"谁提交都全网运行"，而是"谁调用才全网运行"。** 部署只是存代码，调用才是执行代码。

### 读取 vs 写入：谁在跑你的代码？

| 操作 | 谁在执行 | 是否上链 | 是否消耗 Gas | 举例 |
|------|---------|---------|-------------|------|
| **读取（call）** | 只有你连接的节点 | ❌ | ❌ 免费 | 查询代币余额 |
| **写入（transaction）** | 全网所有全节点 | ✅ | ✅ 消耗 | 转账、调用函数 |
| **部署（creation）** | 全网所有全节点 | ✅ | ✅ 消耗（较贵） | 发送新合约到链上 |

## 2. EVM 对合约的硬性限制

EVM 不是通用计算机，它是一个**极度受限的执行环境**。限制不是缺陷，而是为了确保全网节点能高效达成共识。

### 2.1 代码大小：24KB 上限

```
EIP-170: 合约字节码最大 24,576 字节（24KB）

为什么？
  - 每个全节点都要存储所有合约的字节码
  - 如果允许无限大 → 节点存储成本失控 → 中心化
  - 24KB ≈ 1000-1500 行 Solidity 代码

实际影响：
  - 复杂合约必须拆分成多个合约
  - Uniswap V3 核心合约因大小限制被迫精简代码
  - 催生了代理模式（proxy）和合约拆分模式

类比：每个公民只能带 24KB 的行李上飞机
      超过了？拆成多个行李箱（多个合约）
```

### 2.2 计算量：Gas 上限

```
每个区块 Gas 上限：约 30M Gas
每笔交易 Gas 上限：由发送者设定（默认由钱包估算）

这意味着什么？

  ❌ 不能写循环次数不确定的代码
     for (uint i = 0; i < array.length; i++) { ... }
     如果 array 有 10000 个元素 → Gas 用完 → 交易回滚

  ❌ 不能做复杂计算
     训练 AI 模型？不可能。
     复杂排序？Gas 会爆。

  ❌ 不能递归太深
     EVM 调用栈深度限制 1024 层（EIP-150 后实际更低）

类比：你进了一个自助餐厅，但每吃一口都要付钱
      钱花完了（Gas 用完）→ 直接把你赶出去（交易回滚）
      已经吃进去的也算你付了钱（Gas 不退！）
```

### 2.3 操作码：能做什么与不能做什么

```
EVM 禁止的操作：
  ❌ 网络请求（不能发 HTTP 请求）
  ❌ 文件读写（不能访问本地磁盘）
  ❌ 随机数（没有真随机，只有伪随机）
  ❌ 多线程（EVM 是单线程执行）
  ❌ 浮点运算（Solidity 没有 float/double）
  ❌ 系统调用（不能执行 shell 命令）

EVM 可以做的操作：
  ✅ 算术运算（加减乘除，整数）
  ✅ 逻辑运算（比较、与或非）
  ✅ 存储读写（SLOAD/SSTORE）
  ✅ 合约间调用（CALL/DELEGATECALL）
  ✅ 事件日志（LOG0-LOG4）
  ✅ 加密运算（SHA256、ECDSA 等，预编译合约）
```

### 2.4 时间与随机数

```
时间：
  - 合约只能获取 block.timestamp（当前区块时间戳）
  - 精度：秒级
  - 矿工可以在小范围内操纵时间戳（±15秒）

随机数：
  - EVM 是确定性环境 → 同一输入必须同一输出
  - 不能用 block.timestamp 或 blockhash 做随机数（可被矿工操纵）
  - 可靠的随机数必须来自链外（Chainlink VRF 等）

类比：你住在一个没有窗户的房间里
      你不知道外面是白天还是黑夜
      你只能问"看守"（矿工），但他可能骗你
```

## 3. 部署成本详解

### 3.1 直接成本（Gas）

```
部署成本 = 字节码大小 × 每字节 Gas × Gas Price
```

| 合约复杂度 | 字节码大小 | 部署 Gas | L1 成本（20 Gwei） | L2 成本 |
|-----------|----------|---------|-------------------|---------|
| 简单 ERC-20 | ~3KB | 60万-150万 | $60-$150 | $1-$3 |
| 中等复杂度 | ~5KB | 100万-300万 | $120-$300 | $2-$5 |
| 复杂 DeFi | ~20KB | 500万-1000万 | $300-$600 | $5-$15 |
| 代理+逻辑 | ~8KB+8KB | 200万-400万 | $200-$400 | $3-$8 |
| 完整协议（多合约） | ~50KB+ | 1000万-3000万 | $600-$1800 | $10-$30 |

> **注意**：L2 上的部署成本低 90-99%，但 L2 的安全性依赖 L1。

### 3.2 运行成本

| 操作 | Gas | L1 成本 | L2 成本 |
|------|-----|--------|---------|
| 读取状态（view） | 0 | $0 | $0 |
| ERC-20 transfer | ~50,000 | ~$3 | ~$0.01 |
| ERC-20 approve | ~46,000 | ~$2.8 | ~$0.01 |
| Uniswap swap | ~200,000 | ~$12 | ~$0.05 |
| 写入一个存储槽（SSTORE） | 20,000 | ~$1.2 | ~$0.005 |
| 部署新合约 | 1,000,000+ | ~$60+ | ~$0.5+ |

### 3.3 存储成本：链上是世界上最贵的硬盘

```
在以太坊 L1 上存 1KB 数据：
  SSTORE（新写）：20,000 Gas / 32 字节
  1KB = 32 次 SSTORE = 640,000 Gas
  20 Gwei × 640,000 = 0.0128 ETH ≈ $38.4

  AWS S3 存 1KB 数据：$0.000000023

  ────────────────────────
  贵了约 17 亿倍
  ────────────────────────

  设计原则：
    链上只存关键数据（哈希、所有权、余额）
    大文件存在 IPFS/Arweave
    链上做最小信任验证，链下做大量计算和存储
```

### 3.4 隐性成本

| 隐性成本 | 说明 | 金额 |
|---------|------|------|
| **安全审计** | 专业公司审查合约代码 | $15,000 - $500,000+ |
| **形式化验证** | 数学证明合约逻辑正确 | $50,000 - $200,000+ |
| **Bug 悬赏** | 持续奖励发现漏洞的人 | 几千到几百万/年 |
| **不可修改的后果** | bug 被利用后的资金损失 | 无法估量 |
| **Gas 优化开发** | 反复优化节省 Gas 的时间 | 开发者工时 |
| **升级架构** | 代理模式增加的复杂性和审计成本 | 额外审计费用 |

### 3.5 不同链的部署成本对比

| 链 | 部署 ERC-20 合约 | 出块时间 | 安全性 | 适合场景 |
|----|-----------------|---------|--------|---------|
| Ethereum L1 | $60-$150 | 12s | 最高 | 高价值结算 |
| Arbitrum | $2-$5 | <1s | 高（依赖L1） | DeFi 应用 |

## 4. 合约升级：不可修改的代码如何迭代？

智能合约一旦部署就不可修改——这是核心特性，也是最大挑战。但业务需要迭代，bug 需要修复。如何解决？

### 4.1 核心矛盾

```
智能合约的设计：不可修改（Immutable）
业务需求：必须迭代（Iterate）

  ────────────────────────────

  你盖了一栋楼，地基和承重墙永远不能改
  但住户的需求在不断变化
  你需要一种方法：不动承重墙，也能装修房子
```

### 4.2 代理模式（Proxy Pattern）

代理模式是当前最主流的升级方案，核心思路是**分离存储和逻辑**。

```
用户 → 代理合约（存数据、存地址）→ 逻辑合约（存代码、执行逻辑）
              ↑                          ↑
          地址不变                   可以替换
          数据不动                   更新代码
```

```
原理：

  ┌──────────────────────────────┐
  │       代理合约（Proxy）        │
  │                              │
  │  storage:                    │
  │    - implementation 地址      │ ← 指向逻辑合约
  │    - owner                   │
  │    - 所有业务数据             │
  │                              │
  │  fallback:                   │
  │    delegatecall(implementation)│ ← 把调用转发给逻辑合约
  └──────────────────────────────┘
                │
                │ delegatecall
                ↓
  ┌──────────────────────────────┐
  │      逻辑合约（Logic）        │
  │                              │
  │  code:                       │
  │    - transfer()              │
  │    - approve()               │
  │    - 其他业务逻辑             │
  │                              │
  │  注意：逻辑合约的存储布局      │
  │  必须与代理合约兼容！          │
  └──────────────────────────────┘
```

**delegatecall 的关键**：它让逻辑合约的代码在代理合约的存储空间中执行——数据留在代理里，逻辑从逻辑合约来。

### 4.3 主流代理模式对比

| 模式 | 代表 | 升级方式 | 存储安全 | Gas 开销 | 适用场景 |
|------|------|---------|---------|---------|---------|
| **UUPS** | OpenZeppelin | 逻辑合约内升级 | ✅ 安全 | 低 | 通用推荐 |
| **Transparent** | OpenZeppelin | 管理员单独升级 | ✅ 安全 | 较高 | 需要权限分离 |
| **Diamond（EIP-2535）** | EIP-2535 | 多逻辑合约 facet | ⚠️ 复杂 | 中 | 超大合约拆分 |
| **Beacon** | EIP-1967 | 通过 beacon 合约升级 | ✅ 安全 | 中 | 多合约统一升级 |

### 4.4 升级流程与成本

```
升级步骤：

  1. 开发新版本逻辑合约
  2. 部署新逻辑合约（$60-$600，取决于大小）
  3. 调用代理合约的 upgradeTo(新逻辑合约地址)
  4. 用户仍通过代理合约地址交互（地址不变！）

  ────────────────────────────

升级成本：

| 项目 | 成本 | 说明 |
|------|------|------|
| 部署新逻辑合约 | $60-$600 | 每次升级都要部署新合约 |
| upgradeTo 交易 | ~$5-$20 | 更新代理中的实现地址 |
| 重新审计 | $10,000-$200,000+ | 每次升级都应审计 |
| 存储布局冲突修复 | 可能极大 | 如果存储布局不兼容 |

  ────────────────────────────

升级的隐性风险：

  ❌ 存储布局冲突：新逻辑合约的变量顺序必须与旧版本兼容
  ❌ 初始化问题：新逻辑合约的初始化逻辑可能需要特殊处理
  ❌ 权限风险：谁有权升级？中心化风险？
  ❌ 审计成本：每次升级都要重新审计
```

### 4.5 存储布局冲突——升级最容易踩的坑

```solidity
// ❌ 原版合约 V1
contract LogicV1 {
    uint256 public value;      // slot 0
    address public owner;      // slot 1
}

// ❌ 升级版合约 V2 — 在中间插入了新变量！
contract LogicV2 {
    uint256 public value;      // slot 0
    uint256 public newField;   // slot 1 ← 新变量占了原来 owner 的位置！
    address public owner;      // slot 2 ← owner 被挤到了 slot 2
}
// 结果：owner 变量读出来的值是 newField 的值，数据全乱了！

// ✅ 正确做法：只能在末尾追加
contract LogicV2 {
    uint256 public value;      // slot 0
    address public owner;      // slot 1
    uint256 public newField;   // slot 2 ← 只在末尾追加
}
```

> **铁律**：升级合约时，**不能修改已有变量的顺序和类型，只能追加新变量**。

### 4.6 升级权限：谁有权升级？

```
升级权限模型：

1. 中心化升级（Owner）
   owner → upgradeTo(新地址)
   → 风险：单点故障，owner 私钥泄露 = 合约被恶意升级
   → 适合：早期项目

2. 多签升级（MultiSig）
   3/5 多签 → upgradeTo(新地址)
   → 需要 5 个人中 3 个同意才能升级
   → 适合：成熟项目

3. 时间锁升级（Timelock）
   提议升级 → 等待 48 小时 → 执行升级
   → 社区有时间审查和反应
   → 适合：DeFi 协议

4. 治理升级（Governance）
   提案 → 投票 → 通过后升级
   → 最去中心化，但最慢
   → 适合：DAO

5. 不可升级（Immutable）
   没有升级功能
   → 最安全，但 bug 无法修
   → 适合：简单标准合约（如 ERC-20）
```

## 5. 完整成本清单

### 5.1 首次部署总成本

```
小型项目（单个 ERC-20 合约）：
  部署 Gas：$60-$150
  审计：$0（简单合约可自审）
  总计：~$100-$150

中型项目（DeFi 协议，5-10 个合约）：
  部署 Gas：$500-$2,000
  审计：$30,000-$100,000
  Bug 悬赏准备金：$10,000-$50,000
  总计：~$40,000-$150,000

大型项目（完整 DeFi 生态，20+ 合约）：
  部署 Gas：$2,000-$10,000
  审计：$100,000-$500,000+
  Bug 悬赏准备金：$50,000-$500,000+
  形式化验证：$50,000-$200,000
  总计：~$200,000-$1,000,000+
```

### 5.2 迭代升级总成本

```
每次升级的成本：

  部署新逻辑合约：$60-$600
  upgradeTo 交易：$5-$20
  差异审计：$10,000-$100,000
  ────────────────────────────
  单次升级：~$10,000-$100,000+

一年迭代 4 次：$40,000-$400,000+/年

  ────────────────────────────

  这就是为什么"不可修改"是代价也是保护：
  如果升级很贵 → 你会更谨慎地写代码 → bug 更少
  如果升级很便宜 → 你会更随意 → bug 更多
```
