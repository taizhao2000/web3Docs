# 去中心化监控告警体系最佳实践

> 智能合约上线主网只是项目生命周期的起点。DeFi 协议面临着极端行情脱锚、黑客潜伏攻击和基础设施故障等多重威胁。建立一套覆盖“协议层、基础设施层、热钱包余额层”的指标监控告警体系（SecOps），并确立 P0/P1/P2 分级响应响应机制，是保障 Web3 商业项目长治久安的关键。本指南将深度解密生产级 Web3 监控指标红线与告警分级治理最佳实践。

---

## 1. 监控大盘的核心指标红线（Metrics Checklist）

一个成熟的 Web3 项目，其链下监控大盘必须同时对**协议层（业务逻辑）**和**基础设施层（运行环境）**这两大维度的核心指标进行毫秒级轮询：

### 1.1 协议及业务级核心指标 (Protocol-level Metrics)

1. **TVL（总锁仓量）异动波动率**：
   - **监控红线**：TVL 在 1 小时内非正常骤降超过 **$10\%$**。
   - **意义**：这通常是协议正遭受黑客“蚂蚁搬家式”漏洞套取、或者大户由于恐慌发生踩踏式撤资的铁证。
2. **非正常大额赎回（Large Outflow）**：
   - **监控红线**：单笔提现或赎回金额大于 **$\$200,000$**（稳定币或等值代币）。
   - **意义**：必须重点监控该资金流出的目标钱包地址，研判是否为洗钱黑地址或攻击者暂存地址。
3. **价格脱锚度（Price De-peg）**：
   - **监控红线**：协议内挂钩的稳定币（如 USDC/USDT）或流动性质押代币（LST，如 stETH/rETH）与底层标的资产的价格偏差超过 **$1.5\%$**。
   - **意义**：防止套利空间被操纵。
4. **交易滑点与无常损失异常**：
   - **监控红线**：用户交互时的滑损（Slippage）长期处于预设最差滑点边界、或者 StakingPool 的 APR 发生数学突变。

---

### 1.2 基础设施级核心指标 (Infrastructure-level Metrics)

1. **Keeper / Relayer 热钱包 Gas 余额（最易发生的空窗惨剧）**：
   - **监控红线**：负责自动化执行清算、预言机喂价、或代签交易（Relayer）的热钱包 EOA 账户 ETH 余额**低于 $0.5$ ETH**。
   - **严重性**：这是 Web3 最经典的低级故障。由于热钱包 ETH 耗尽，导致清算交易在发生行情暴跌时因“余额不足（Insufficient Funds）”无法发出，进而导致协议产生巨额穿仓坏账。必须设置低于 $0.5$ ETH 立刻触发充值提醒。
2. **RPC 节点延迟（Latency）与错误率**：
   - **监控红线**：WebSocket / HTTP 节点响应延迟**大于 2000 毫秒**，或者 RPC 连续返回 5xx 错误、连接超时错误率**大于 $5\%$**。
   - **意义**：RPC 故障会导致所有的监听器死掉、所有的交易广播失败。必须配置 **FallbackProvider（热备节点冗余机制）** 并在主节点故障时瞬时自动切换。
3. **区块脱节监控（Block Lag）**：
   - **监控红线**：本地节点或监听服务读取到的最新区块高度，落后于以太坊主网真实区块高度**超过 3 个区块**。
   - **意义**：防止因为读取到脏区块/分叉孤块（Reorg）数据导致监控产生严重的延迟漏报。

## 2. 监控告警的分级响应机制（On-call Alerting Tiering）

并不是所有的警报都需要半夜叫醒开发人员。建立科学的 **P0 / P1 / P2 告警分级治理机制**，可以让团队聚焦在真正致命的威胁上：

```
                 【 链上/链下监控事件触发 】
                             │
         ┌───────────────────┼───────────────────┐
         ▼                   ▼                   ▼
     【 P0 级 】         【 P1 级 】         【 P2 级 】
   (协议生死存亡)       (高危逻辑异动)       (日常业务日志)
         │                   │                   │
         ▼                   ▼                   ▼
    一键自动熔断        发送 Slack/TG       发送常规邮件
  电话轰炸 PagerDuty   30分钟内人工接入     每日日终审计
```

### 🚨 P0 级：灭顶之灾（CRITICAL - 5分钟内响应）
- **触发场景**：
  - 合约重入锁（Reentrancy Guard）触发失败。
  - 合约的所有权（Owner）或者多签管理员发生非预期变动。
  - 核心池 TVL 在 10 分钟内暴跌 $20\%$ 以上。
  - 负责清算的 Keeper 钱包 Gas 归零。
- **响应动作**：
  1. **自动化联动**：Sentinel 或哨兵脚本立即在链上抢跑发起交易，**将协议一键熔断暂停（Pause）**，终止所有的 deposit 和 borrow。
  2. **人工通知**：通过 PagerDuty、Opsgenie 平台，对所有的 On-call 核心安全与开发团队成员进行**电话电话轰炸（Phone Call）、24小时短信（SMS）轰炸**，必须在 5 分钟内有人接入排查。

### ⚠️ P1 级：高危异常（WARNING - 30分钟内响应）
- **触发场景**：
  - 多签钱包（Multi-Sig）内提交了新的修改提案。
  - 预言机（Chainlink）喂价数据超过 45 分钟未更新（Stale Price）。
  - 单笔资金赎回超过 $\$500,000$ 的大户出金行为。
  - RPC 节点延迟飙升，开始触发自动主备切换。
- **响应动作**：
  - 发送加粗高亮警报至 Slack、Telegram 的安全监控频道（Alerter Channel）。
  - 相关安全开发值班人员必须在 30 分钟内登录后台并对日志进行排查，确认是否需要升级为 P0 或进行常规技术干预。

### ℹ️ P2 级：常规日志（INFO - 24小时内审计）
- **触发场景**：
  - 例行多签提案执行成功。
  - 每日业务收益日报、费用提取、例行小额取款。
  - Keeper 钱包余额低于 1 ETH 的常规补款提示。
- **响应动作**：
  - 发送至日常运营频道、邮件日报，由运营团队在日终进行统一审阅和账目对账。

## 3. Prometheus + Grafana 一站式 Web3 监控架构

在大型 Web3 商业团队中，推荐使用工业级的监控大盘方案：

```
 ┌────────────────┐
 │  以太坊主网 /  │ 
 │  Layer2 节点   │
 └────────────────┘
         │
         ▼ (JSON-RPC)
 ┌────────────────┐
 │ Prometheus     │ ────────> 【 Prometheus 数据库 】
 │ Exporter 导出器│           (存储 TVL、Gas、余额历史时序数据)
 └────────────────┘                           │
                                              ▼
                                     ┌────────────────┐
                                     │  Grafana 大盘  │ (可视化图表)
                                     └────────────────┘
                                              │
                                              ▼ (告警触发)
                                     ┌────────────────┐
                                     │   PagerDuty    │ (电话/短信警报)
                                     └────────────────┘
```

1. **Prometheus Exporter**：编写自定义的 Node.js 脚本或使用现有的区块链 Exporter。每 15 秒通过 JSON-RPC 读取合约的 `balanceOf`、`totalSupply` 和 Keeper 钱包的 `eth_getBalance`。
2. **Prometheus 时序数据库**：将读取到的 TVL、Gas 余额、转账频次以时间序列（Time Series）形式存入 Prometheus 中。
3. **Grafana 可视化配置**：配置炫酷、直观的监控 Dashboard（通常在办公室大屏幕投屏），设定触发阈值红线。
4. **联动 PagerDuty 报警**：一旦时序数据跌破红线，Grafana 自动推送 API 到 PagerDuty 实施电话警报，实现 Web3 项目的一站式监控告警闭环。
