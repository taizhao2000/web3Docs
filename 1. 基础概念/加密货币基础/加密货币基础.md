## 1. 什么是加密货币？

**一句话定义**：

加密货币是基于密码学原理（主要是非对称加密和哈希函数）确保安全的数字资产，运行在去中心化区块链网络上，用于价值传输和激励机制。

**与法币的关键区别**：

| 维度 | 法币（如人民币、美元） | 加密货币（如 BTC、ETH） |
|------|---------------------|----------------------|
| 发行方 | 中央银行 | 协议规则（算法） |
| 信任基础 | 政府信用 | 密码学 + 共识机制 |
| 供应量 | 可无限增发 | 多数有上限或可预测 |
| 转账中介 | 银行/支付机构 | 无需中介（点对点） |
| 审查性 | 可冻结/扣押 | 不可冻结（掌握私钥即拥有） |
| 交易可逆性 | 支持退款/冲正 | 不可逆（一旦确认无法撤销） |

## 2. 代币（Token）

### 2.1 原生代币 vs 应用代币

| 类型 | 定义 | 例子 | 用途 |
|------|------|------|------|
| **原生代币（Native Token）** | 区块链协议本身发行的资产，用于支付 Gas 和激励 | BTC、ETH、SOL | 网络安全 + 交易手续费 |
| **应用代币（App Token）** | 在智能合约中发行的资产，运行在已有链上 | USDT、UNI、AAVE | 治理、功能、价值捕获 |

**关键区别**：原生代币是链运行的**刚需**——没有 ETH 就无法在 Ethereum 上发起交易；没有应用代币，链照样运转。

### 2.2 代币标准详解

#### ERC-20（同质化代币）

最广泛使用的代币标准，每个代币与另一个代币完全等价（1 USDT = 1 USDT）。

```solidity
// ERC-20 核心接口
interface IERC20 {
    function totalSupply() external view returns (uint256);           // 总供应量
    function balanceOf(address account) external view returns (uint256); // 余额查询
    function transfer(address to, uint256 amount) external returns (bool); // 转账
    function allowance(address owner, address spender) external view returns (uint256); // 授权额度
    function approve(address spender, uint256 amount) external returns (bool); // 授权
    function transferFrom(address from, address to, uint256 amount) external returns (bool); // 授权转账
}
```

**为什么需要 approve + transferFrom？**

这是 DeFi 的基础交互模式——用户先授权（approve）合约可以动用自己的代币，合约再通过 `transferFrom` 执行实际转移。这允许合约在单笔交易中完成多步操作。

```
用户 → approve(Dex合约, 1000 USDT)    // 授权
用户 → Dex.swap(1000 USDT → ETH)       // 触发交易
  ↓
Dex合约内部调用：
  USDT.transferFrom(用户, Dex, 1000)   // Dex 拿走 USDT
  WETH.transfer(用户, 0.5 ETH)          // Dex 给用户 ETH
```

#### ERC-721（非同质化代币）

每个代币拥有唯一 ID，不可互换。

```solidity
interface IERC721 {
    function ownerOf(uint256 tokenId) external view returns (address);  // 查询所有者
    function safeTransferFrom(address from, address to, uint256 tokenId) external; // 安全转移
    function approve(address to, uint256 tokenId) external;            // 授权
}
```

#### ERC-1155（多代币标准）

同一合约管理多种代币（同质化 + 非同质化），支持批量操作。

```solidity
interface IERC1155 {
    function balanceOf(address account, uint256 id) external view returns (uint256);
    function safeBatchTransferFrom(
        address from, address to,
        uint256[] calldata ids,
        uint256[] calldata amounts,
        bytes calldata data
    ) external;  // 批量转移
}
```

**三种标准对比**：

| 特性 | ERC-20 | ERC-721 | ERC-1155 |
|------|--------|---------|----------|
| 代币类型 | 同质化 | 非同质化 | 混合 |
| 批量转账 | 不支持 | 不支持 | 原生支持 |
| 单合约代币数 | 1 种 | 1 种 | 无限种 |
| Gas 效率 | 中 | 低 | 高 |
| 适用场景 | 货币、治理 | 艺术品、身份 | 游戏、NFT+FT 混合 |

### 2.3 代币经济学核心指标

| 指标 | 英文 | 说明 |
|------|------|------|
| **流通供应量** | Circulating Supply | 当前市场上可交易的代币数量 |
| **总供应量** | Total Supply | 已发行的所有代币（含锁仓部分） |
| **最大供应量** | Max Supply | 代币总量上限（如 BTC = 2100万） |
| **市值** | Market Cap | 流通供应量 × 当前价格 |
| **完全稀释市值** | FDV (Fully Diluted Valuation) | 最大供应量 × 当前价格 |
| **通胀率** | Inflation Rate | 每年新增代币占总供应量的比例 |

**为什么要看 FDV 而不是市值？**

很多项目早期流通量极低（如总量的 5%），市值看起来很小，但 FDV 可能已经极高。如果所有代币解锁后抛售，价格会大幅下跌。

```
示例：
流通量 = 5000万（总量的 5%）
价格 = $10
市值 = $5亿      ← 看起来合理
FDV = $100亿     ← 解锁后抛压巨大
```

## 3. 钱包（Wallet）

### 3.1 钱包的本质

> **钱包不存储代币。代币在链上。钱包存储的是私钥。**

钱包的核心功能是**密钥管理**——安全地生成、存储、使用私钥来签名交易。

```
钱包不持有资产
    ↓
私钥 → 签名 → 授权资产转移
    ↓
谁掌握私钥，谁就拥有资产
    ↓
"Not your keys, not your coins"
```

### 3.2 密钥生成与派生

#### 单密钥生成（基础方式）

```
1. 生成 256 位随机数 → 私钥
2. 私钥 → secp256k1 椭圆曲线 → 公钥
3. 公钥 → Keccak256 哈希 → 取后 20 字节 → 地址
```

#### HD 钱包（分层确定性钱包）

大多数现代钱包使用 **BIP-32/BIP-39/BIP-44** 标准的 HD 钱包，从一组助记词派生无数个密钥。

```
助记词（12/24 个英文单词）
    ↓ BIP-39
种子（Seed，512 bits）
    ↓ BIP-32
主私钥 + 主链码
    ↓ BIP-32 派生
子密钥（按路径索引派生，可无限扩展）
    ↓ BIP-44 路径规范
m/44'/60'/0'/0/0  ← Ethereum 第一个账户
m/44'/60'/0'/0/1  ← Ethereum 第二个账户
m/44'/0'/0'/0/0   ← Bitcoin 第一个账户
```

**BIP-44 路径格式**：`m / purpose' / coin_type' / account' / change / address_index`

| 字段 | 含义 | Ethereum 示例 |
|------|------|--------------|
| purpose | BIP 标准版本 | 44' |
| coin_type | 币种编号 | 60'（ETH）、0'（BTC） |
| account | 账户索引 | 0' |
| change | 找零（BTC 用） | 0 |
| address_index | 地址索引 | 0, 1, 2... |

**HD 钱包的优势**：

- 一组助记词恢复所有地址
- 无需备份每个私钥
- 可在不联网的情况下生成新地址（观察钱包）

### 3.3 钱包类型对比

| 类型 | 热钱包 | 冷钱包 | 合约钱包 |
|------|--------|--------|---------|
| **联网状态** | 始终在线 | 始终离线 | 在链上 |
| **安全性** | 中 | 高 | 取决于实现 |
| **便利性** | 高 | 低 | 中 |
| **代表** | MetaMask、Trust Wallet | Ledger、Trezor | Gnosis Safe、Argent |
| **私钥存储** | 设备/浏览器 | 专用硬件芯片 | 无单点私钥 |
| **多签支持** | 否 | 否 | 是 |
| **社交恢复** | 否 | 否 | 是 |
| **适用场景** | 日常小额 | 大额长期持有 | DAO、团队金库 |

### 3.4 签名与交易授权流程

```
1. 构建交易（to, value, data, gas, nonce...）
    ↓
2. 序列化交易（RLP 编码）
    ↓
3. 计算交易哈希（Keccak256）
    ↓
4. 用私钥签名（ECDSA/secp256k1）
   → 生成 v, r, s 三个值
    ↓
5. 附加签名到交易数据
    ↓
6. 广播已签名交易到网络
    ↓
7. 节点用公钥（从 v,r,s 推导）验证签名
```

**注意**：签名过程在钱包内完成，**私钥永远不会离开钱包**。网络验证者只需要公钥即可验证签名有效性。

## 4. 交易（Transaction）

### 4.1 交易的本质

交易是区块链上**状态变更的唯一方式**。每笔交易都是一个签名授权的指令，指示区块链从一个状态转换到另一个状态。

```
状态 A（交易前）──[交易]──→ 状态 B（交易后）

例：
Alice 余额 10 ETH, Bob 余额 5 ETH
    ──[Alice→Bob 2 ETH]──→
Alice 余额 8 ETH, Bob 余额 7 ETH
```

### 4.2 Ethereum 交易结构

#### Legacy 交易（Type 0）

```javascript
{
    nonce: 7,                    // 发送者已发送的交易数
    gasPrice: 20000000000,       // Gas 价格（20 Gwei）
    gasLimit: 21000,             // Gas 上限
    to: "0xABC...DEF",          // 接收地址
    value: 1000000000000000000,  // 转账金额（1 ETH = 10^18 wei）
    data: "0x",                  // 附加数据（合约调用时使用）
    v: 27,                       // 签名恢复标识
    r: "0x123...abc",            // 签名 r 值
    s: "0x456...def"             // 签名 s 值
}
```

#### EIP-1559 交易（Type 2，推荐）

```javascript
{
    chainId: 1,                  // 链 ID（防重放攻击）
    nonce: 7,
    maxPriorityFeePerGas: 2000000000,   // 给矿工的小费（2 Gwei）
    maxFeePerGas: 50000000000,          // 愿意支付的最高 Gas 价格（50 Gwei）
    gasLimit: 21000,
    to: "0xABC...DEF",
    value: 1000000000000000000,
    data: "0x",
    accessList: [],              // EIP-2930 预热地址列表
    signature: { v, r, s }
}
```

**EIP-1559 费用机制**：

```
实际 Gas Price = min(maxFeePerGas, baseFee + maxPriorityFeePerGas)
矿工收入 = maxPriorityFeePerGas × gasUsed
销毁 = baseFee × gasUsed（从流通中永久移除）

           用户支付
          ┌──────────────────────────┐
          │                          │
          │   ┌─────────────┐         │
          │   │ baseFee     │  ← 销毁 │
          │   ├─────────────┤         │
          │   │ priorityFee │  ← 矿工│
          │   └─────────────┘         │
          │                          │
          └──────────────────────────┘
```

| 字段 | 说明 |
|------|------|
| baseFee | 协议自动计算，与网络拥堵程度正相关，每区块可增减 12.5% |
| maxPriorityFeePerGas | 用户给矿工的小费，竞争打包优先级 |
| maxFeePerGas | 用户愿意支付的最高 Gas 单价（含 baseFee + 小费） |

### 4.3 交易生命周期

```
1. 构建交易
    ↓
2. 钱包签名
    ↓
3. 广播到网络（通过 RPC 节点）
    ↓
4. 进入 Mempool（待处理交易池）
    ↓
5. 矿工/验证者选取并执行
    ↓
6. 打包进区块
    ↓
7. 广播新区块
    ↓
8. 网络验证并接受
    ↓
9. 等待足够确认数 → 交易最终确认
```

### 4.4 交易类型

| 类型 | to 字段 | data 字段 | 说明 |
|------|---------|----------|------|
| **普通转账** | EOA 地址 | "0x"（空） | ETH 转账，Gas 固定 21000 |
| **合约部署** | null（空） | 合约字节码 | 部署新合约，Gas 较高 |
| **合约调用** | 合约地址 | 函数选择器+参数 | 调用合约方法，Gas 不定 |

```javascript
// 普通转账
tx = { to: "0xABC...", value: parseEther("1.0"), data: "0x" }

// 合约部署
tx = { to: null, data: contractBytecode }

// 合约调用（ERC-20 transfer）
tx = {
    to: tokenAddress,
    data: "0xa9059cbb" + encodeParams(recipient, amount)  // transfer(address,uint256)
}
```

## 5. Gas 机制

### 5.1 什么是 Gas？

Gas 是衡量链上计算和存储资源消耗的单位。每一步 EVM 操作都有固定的 Gas 成本。

> **类比**：Gas 就像出租车计价器——上车有起步价，每走一步都要付费。复杂的操作（合约调用）就像长途旅行，简单的操作（ETH 转账）就像短途。

### 5.2 常见操作的 Gas 成本

| 操作 | Gas 成本 | 说明 |
|------|---------|------|
| ETH 转账 | 21,000 | 固定值 |
| ERC-20 转账 | ~50,000-65,000 | 调用合约的 transfer 方法 |
| ERC-20 approve | ~46,000 | 授权操作 |
| Uniswap V2 交易 | ~150,000-200,000 | 涉及多合约调用 |
| Uniswap V3 交易 | ~200,000-300,000 | 更复杂的逻辑 |
| 合约部署 | 数十万~数百万 | 取决于合约复杂度 |
| SSTORE（新写入） | 20,000 | 写入存储槽（最贵的操作之一） |
| SLOAD（读取） | 2,100 | 从存储读取 |
| LOG（事件） | 375 + 375×topics + 8×bytes | 事件日志 |

### 5.3 Gas 费用计算

```
Legacy 交易:
  费用 = gasUsed × gasPrice

EIP-1559 交易:
  费用 = gasUsed × effectiveGasPrice
  其中 effectiveGasPrice = min(maxFeePerGas, baseFee + maxPriorityFeePerGas)
```

**实际例子**：

```
一笔 ERC-20 转账：
  gasUsed = 55,000
  baseFee = 15 Gwei
  maxPriorityFeePerGas = 2 Gwei
  effectiveGasPrice = 15 + 2 = 17 Gwei
  费用 = 55,000 × 17 Gwei = 935,000 Gwei = 0.000935 ETH
  （ETH 价格 $3,000 时 ≈ $2.81）
```

### 5.4 Gas 优化要点

| 优化方向 | 方法 | 节省幅度 |
|---------|------|---------|
| **时间选择** | 在低 Gas 时段交易（周末/凌晨 UTC） | 30-70% |
| **Gas 预估** | 使用 eth_gasPrice 而非固定值 | 避免过高出价 |
| **EIP-1559** | 使用 Type 2 交易，设置合理的 maxFee | 避免多付 |
| **批量操作** | 一次交易完成多步操作 | 节省基础 Gas |
| **合约优化** | 减少 SSTORE、使用 calldata | 20-50% |
| **Layer 2** | 在 L2 上执行交易 | 90-99% |

## 6. Nonce 机制

### 6.1 什么是 Nonce？

Nonce 是每个账户的**交易计数器**，从 0 开始，每发送一笔交易 +1。

```
Alice 的交易历史：
  Nonce 0: 部署合约
  Nonce 1: 转账给 Bob
  Nonce 2: 参与 ICO
  Nonce 3: ...  ← 下一笔交易的 nonce 必须是 3
```

### 6.2 Nonce 的三个核心作用

| 作用 | 说明 |
|------|------|
| **防重放** | 同一 Nonce 的交易只能执行一次，防止已广播的交易被重复提交 |
| **交易排序** | 矿工按 Nonce 顺序打包，保证交易执行顺序与发送顺序一致 |
| **防双花** | 同一 Nonce 只能有一笔交易被确认，从账户层面防止双花 |

### 6.3 Nonce 的规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **严格递增** | Nonce 必须从 0 开始，逐个递增 | Nonce 0, 1, 2, 3... |
| **不可跳跃** | Nonce 3 未确认时，Nonce 4 不会被处理 | 发送 Nonce 4 时会卡在 Mempool |
| **替代机制** | 同一 Nonce 的更高 Gas 价格交易可替换旧交易 | 加速未确认的交易 |
| **并发限制** | 同一地址不能同时发送多笔交易 | 必须等上一笔确认 |

### 6.4 Nonce 的坑与最佳实践

```javascript
// ❌ 常见错误：快速连续发送交易
await contract.transfer(alice, 100);  // Nonce 5
await contract.transfer(bob, 200);     // Nonce 5 ← 可能失败！前一笔还没确认

// ✅ 正确方式 1：等确认
const tx1 = await contract.transfer(alice, 100);
await tx1.wait();  // 等 Nonce 5 确认
const tx2 = await contract.transfer(bob, 200);  // Nonce 6

// ✅ 正确方式 2：手动设置 Nonce
const nonce = await provider.getTransactionCount(signer.address);
await contract.transfer(alice, 100, { nonce: nonce });
await contract.transfer(bob, 200, { nonce: nonce + 1 });
```

**Nonce 跳跃问题**：

```
已确认: Nonce 0, 1, 2
Mempool: Nonce 4 ← 跳过了 3！
结果: Nonce 4 永远不会被处理，直到 Nonce 3 被发送并确认
```

## 7. 地址系统

### 7.1 地址生成流程

```
随机数（256 bits）
    ↓ secp256k1 椭圆曲线
私钥（64 hex = 32 bytes）
    ↓ 椭圆曲线点乘
公钥（128 hex = 64 bytes，未压缩格式）
    ↓ Keccak-256 哈希
哈希值（64 hex = 32 bytes）
    ↓ 取后 20 字节
地址（40 hex = 20 bytes）
    ↓ EIP-55 校验（可选）
Checksum 地址（混合大小写，如 0xd8dA6BF26964aF9D7eEd9e03E53415D37aA96045）
```

### 7.2 EIP-55 校验地址

Ethereum 地址本不区分大小写，但混合大小写的校验地址可以**检测输入错误**。

```javascript
// 没有校验的地址（全小写）
0xd8da6bf26964af9d7eed9e03e53415d37aa96045

// 有校验的地址（EIP-55）
0xd8dA6BF26964aF9D7eEd9e03E53415D37aA96045
//  ↑大写     ↑大写 ↑大写  ↑大写    ↑大写 ↑大写
```

校验规则：对地址做 Keccak256 哈希，如果哈希对应位 >= 8，则该字符大写。这样输入错误一个字符，校验就会失败。

### 7.3 不同链的地址格式

| 链 | 格式 | 示例 |
|----|------|------|
| Ethereum | 0x + 40 hex | 0xd8dA6BF26964aF9D7eEd9e03E53415D37aA96045 |
| Bitcoin (Legacy) | Base58Check，1 开头 | 1A1zP1eP5QGefi2DMPTfTL5SLmv7DivfNa |
| Bitcoin (SegWit) | Bech32，bc1 开头 | bc1qar0srrr7xfkvy5l643lydnw9re59gtzzwf5mdq |
| Solana | Base58，32-44 字符 | 7xKXtg2CW87d97TXJSDpbD5jBkheTqA83TZRuJosgAsU |
| Cosmos | Bech32，cosmos1 开头 | cosmos1t5v0mq8xd... |

**EVM 兼容链共享以太坊地址格式**：Polygon、Arbitrum、Optimism、BSC 等都使用 `0x` 前缀的 20 字节地址，且同一私钥在所有 EVM 链上派生出相同地址。

## 8. 价值单位与换算

### 8.1 Ethereum 单位体系

| 单位 | 换算关系 | 用途 |
|------|---------|------|
| **wei** | 1 wei = 1 | 最小单位，Gas 计价 |
| **Gwei** | 1 Gwei = 10⁹ wei | Gas 价格报价 |
| **ETH** | 1 ETH = 10¹⁸ wei | 日常使用 |

```
1 ETH = 1,000,000,000,000,000,000 wei
      = 1,000,000,000 Gwei
      = 10^18 wei
```

### 8.2 Bitcoin 单位体系

| 单位 | 换算关系 | 用途 |
|------|---------|------|
| **satoshi（聪）** | 1 sat = 1 | 最小单位 |
| **bit** | 1 bit = 100 sat | 小额支付 |
| **mBTC** | 1 mBTC = 0.001 BTC | 中等金额 |
| **BTC** | 1 BTC = 100,000,000 sat | 日常使用 |

### 8.3 代码中的数值处理

```javascript
// ❌ 错误：JavaScript 的 Number 精度不够
const amount = 0.1 * 10 ** 18;  // 99999999999999992 ← 精度丢失！

// ✅ 正确：使用 ethers.js 的工具
const amount = ethers.parseEther("0.1");     // 100000000000000000n (BigInt)
const fee = ethers.parseUnits("15", "gwei"); // 15000000000n

// ✅ 显示时格式化
ethers.formatEther(amount);  // "0.1"
ethers.formatUnits(fee, "gwei");  // "15.0"
```

> **核心原则**：链上所有数值都是**整数**（wei 为单位），浮点数只用于显示。代码中必须使用 BigInt 或大数库。

## 9. 代币经济学设计

### 9.1 供应模型

| 模型 | 代表 | 机制 | 通胀/通缩 |
|------|------|------|----------|
| **固定供应** | BTC | 总量 2100 万，每 4 年减半 | 通缩（丢失不可恢复） |
| **EIP-1559 销毁** | ETH | baseFee 被销毁，抵消增发 | 可通缩可通胀 |
| **通胀模型** | DOT | 每年固定增发率 | 通胀（质押收益抵消） |
| **算法稳定币** | DAI | 抵押品支撑，动态调整 | 供应随需求变化 |

### 9.2 Bitcoin 减半周期

```
创世区块（2009）: 50 BTC/区块
第1次减半（2012）: 25 BTC/区块
第2次减半（2016）: 12.5 BTC/区块
第3次减半（2020）: 6.25 BTC/区块
第4次减半（2024）: 3.125 BTC/区块
...
约 2140 年: 挖矿奖励趋近于 0，矿工完全依靠手续费
```

### 9.3 Ethereum 的 Ultra Sound Money

EIP-1559 引入 baseFee 销毁机制后，ETH 具备了「可通缩」特性：

```
ETH 增发量 ≈ 0.5%/年（PoS 质押奖励）
ETH 销毁量 ≈ 取决于网络活跃度

当 销毁 > 增发 → ETH 变为通缩
当 销毁 < 增发 → ETH 仍为通胀

2021-2022 期间：ETH 多次出现通缩日/周
```

### 9.4 常见代币分配模型

| 分配对象 | 典型比例 | 锁仓期 |
|---------|---------|--------|
| 团队/创始人 | 15-25% | 1-4 年线性释放 |
| 投资人 | 10-20% | 1-2 年锁仓 |
| 生态基金 | 20-40% | 按用途逐步释放 |
| 社区/空投 | 5-15% | 部分即时，部分线性 |
| 流动性挖矿 | 10-20% | 按区块释放 |

> **投资者提示**：关注解锁时间表（Vesting Schedule），大量代币解锁往往会带来抛压。
